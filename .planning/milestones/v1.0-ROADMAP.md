## Overview

The `add-service` AI skill — an interactive guide that drops production-ready Docker Compose and Taskfile configurations into any project via a single natural-language conversation. Supports 5 services (Redis, RabbitMQ, PostgreSQL, MySQL, MongoDB) plus optional Monitoring stack, with full merge detection, multi-instance alias support, and idempotent re-runs.

## Milestone Summary

**Key Decisions:**

- Decision: `.devtools/` as output directory — Isolated from project source, easy to gitignore or include selectively. Confirmed Phase 01.
- Decision: Per-service Taskfiles included from root — Modular, adding a service doesn't require editing existing files. Confirmed Phase 01.
- Decision: `{{PLACEHOLDER}}` token format — Simple, unambiguous substitution; no shell expansion collisions. Confirmed Phase 01.
- Decision: Three-tier Docker networks (per-service / _services / _monitoring) — Isolation + cross-service reach without exposing internals. Confirmed Phase 01.
- Decision: Plain volume key names (no `${VAR}` in keys) — Docker Compose v2 does not support variable expansion in volume key names. Confirmed Phase 01.
- Decision: Interactive config before writing — Avoids wrong defaults baked in; user owns their config from the start. Confirmed Phase 02.
- Decision: Support both Copilot CLI and Claude/MCP — Maximize portability across AI tools. Confirmed Phase 03.
- Decision: Subfolder per-service layout (`.devtools/<slug>/`) — Supports multi-instance aliases cleanly; each instance has its own `.env`. Adopted Phase 04.
- Decision: Proactive alias prompt on first install — Reduces re-run friction; user sets instance name upfront. Adopted Phase 05.
- Decision: 6 base service names as idempotency guard — Distinguishes "base service re-install" (→ Step 1a) from "alias slug re-run" (→ early exit). Adopted Phase 07.

**Issues Resolved:**
