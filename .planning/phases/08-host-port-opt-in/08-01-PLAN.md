<objective>
Implement the host-port opt-in gate in `skills/add-service/SKILL.md` so that users can choose
whether their service is exposed on a host port, and the compose output exactly matches that choice.

Purpose: Developers often want services reachable only within the Docker network (e.g. a Redis
instance consumed only by an API container). Writing a host port binding when it's not needed
exposes the service on localhost unnecessarily. This phase makes host-port binding opt-in.

Output: `skills/add-service/SKILL.md` updated across four locations:
1. Step 5 — new host-port opt-in question inserted before the Port question
2. Step 9a — in-memory port-line removal when opted out
3. Step 10a — conditional port env var skip when opted out
4. Step 8 — Port row in the summary table reflects opt-out/opt-in state
</objective>

service container:**

**Scope of this modification — do NOT touch:**
- UI companion container `ports:` blocks (redisinsight, pgadmin, phpmyadmin, mongo_express).
  These are ALWAYS host-mapped regardless of opt-out — the browser needs host access. (D-11)
- Monitoring exporter containers (`redis_exporter`, `rabbitmq_exporter`, etc.) — they already
  have no `ports:` block by design. (D-13)

Apply all modifications to the in-memory content before writing — never modify the fetched
template file. (D-10)
