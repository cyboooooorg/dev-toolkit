### Volume naming rule (CRITICAL — Docker Compose limitation)
Volume keys MUST be plain names. `${VAR}_data:` is INVALID at the top-level volumes block.
CORRECT:   `redis_data:`        (Docker auto-prefixes with COMPOSE_PROJECT_NAME at runtime)
INCORRECT: `${COMPOSE_PROJECT_NAME}_redis_data:`  (Docker Compose rejects this)

### Network naming rule (CORRECT — variable expansion works in name: property)
CORRECT:   `name: ${COMPOSE_PROJECT_NAME:-devtools}_redis`  (works in name: property value)
INCORRECT: `${COMPOSE_PROJECT_NAME}_redis:`  (key, not value — doesn't work)

### services network rule (CRITICAL — no external: true)
The `services` network MUST be declared as a regular bridge network in each compose file.
`external: true` causes `docker compose up` to fail if the network doesn't pre-exist.
Docker Compose is idempotent across duplicate named network declarations in included files.

### Profiles rule
Services with `profiles: [services]` only start when COMPOSE_PROFILES=services is in .env.
Do NOT use `external:` on the services network — declare it inline in every compose file.

### Credential rule (CRITICAL)
Compose YAML MUST use `${ENV_VAR}` references, NEVER `{{TOKEN}}` syntax.
`{{TOKEN}}` syntax appears ONLY in .env.example and metadata.json (for the skill to substitute).
Every service container with credentials MUST have `env_file: - .env`.

### No `version:` key
Docker Compose v2 spec: `version:` is obsolete. MUST NOT appear in any compose file.

### Three-tier network model per service
1. Per-service net:  name: ${COMPOSE_PROJECT_NAME:-devtools}_<service>  → service + UI companion
2. App-access net:   name: ${COMPOSE_PROJECT_NAME:-devtools}_services    → main service only
3. Monitoring net:   name: ${COMPOSE_PROJECT_NAME:-devtools}_monitoring  → exporter + Prometheus
